## 🎯 目标

设计并实现一个微信小程序 Demo：

> **门店 / 活动运营助手**

用于展示 **wedux-ui** 在真实业务场景中的使用方式。

该 Demo 必须满足：

* 看起来像真实可上线产品（不是组件展示页）
* 覆盖核心业务流程（浏览 → 创建 → 查看 → 管理）
* 最大化使用组件库能力
* UI 风格统一、可切换主题

---

## 🧩 产品抽象（先理解再实现）

这是一个帮助“商家或运营人员”管理活动的小程序，核心流程：

1. 查看运营数据（首页）
2. 浏览活动列表
3. 创建 / 编辑活动（重点）
4. 查看活动详情并执行操作
5. 个性化设置（主题 / 风格）
6. 生成分享二维码

👉 AI 在实现时必须围绕这个**完整闭环流程**

---

## 📱 页面级能力要求（不是结构，是职责）

### 🏠 首页（Dashboard）

目标：展示“有业务价值的信息密度”

必须体现：

* 数据概览（数字类）
* 快捷操作入口
* 最近内容列表
* 明确的主操作入口（如“新建活动”）

👉 强调“信息 + 操作结合”，不是纯展示

---

### 📋 列表页（List）

目标：模拟真实后台列表体验

必须体现：

* 多状态筛选（如进行中 / 已结束）
* 长列表加载（分页或无限滚动）
* 信息层级清晰（标题 / 状态 / 时间）

👉 避免做成“简单重复列表”

---

### 🧾 表单页（Form）⭐最重要

目标：体现组件库的“生产力”

必须体现：

* 多类型输入组合（文本 / 选择 / 时间 / 上传等）
* 表单结构清晰（分组 or 分段）
* 校验机制（必填 / 格式）
* 提交过程反馈（loading / 成功）

👉 这是决定开发者是否使用库的关键页面

---

### 📄 详情页（Detail）

目标：信息展示 + 操作聚合

必须体现：

* 信息结构化展示
* 用户或参与者信息
* 操作入口（编辑 / 删除 / 更多）

👉 要有“可操作性”，不是纯展示页

---

### ⚙️ 设置页（Theme / Profile）⭐核心卖点页

目标：突出 wedux-ui 差异化能力

必须体现：

* 主题切换（亮 / 暗）
* 主色动态修改
* UI 实时响应变化

👉 这是“说服开发者”的关键页面之一

---

### 🔗 二维码页（Share）

目标：体现业务闭环

必须体现：

* 可生成分享二维码
* 搭配操作按钮（保存 / 分享）

---

## 🧱 组件使用策略（强约束）

AI 必须遵守：

### 1️⃣ 表单相关组件必须系统性使用

不是“用一个 input 就完了”，而是：

* Form + FormItem 结构
* 多种输入组件组合
* 有校验 + 有状态反馈

---

### 2️⃣ 列表必须具备“真实业务感”

必须包含：

* 状态标识（Tag / Badge）
* 文本溢出处理（Ellipsis）
* 加载行为（InfiniteScroll）

---

### 3️⃣ 反馈类组件必须出现

至少覆盖：

* 操作反馈（成功 / 失败）
* 浮层交互（Drawer / Popover / Tooltip）

---

### 4️⃣ 至少一次“组合组件使用”

例如：

* 表单 + 上传 + 日期选择
* 列表 + 无限滚动 + 状态标签

👉 体现组件之间协作能力

---

### 5️⃣ 必须使用主题能力（强制）

必须实现：

* 全局主题控制
* 至少一个页面实时响应主题变化
* 使用 CSS 变量进行样式控制

---

## 🎨 UI / 体验约束

AI 在生成 UI 时必须遵守：

* 统一圆角体系（不要混乱）
* 统一间距（8 / 12 / 16 规则）
* 主色一致
* 信息分层清晰（标题 > 描述 > 辅助）
* 避免组件堆砌（要有布局逻辑）

---

## 🔁 状态设计（必须体现）

每类页面至少包含：

* Loading 状态
* 空状态
* 正常状态
* 操作反馈状态

---

## 🧠 数据策略

* 使用本地 mock 数据
* 数据结构需贴近真实业务（不要过度简化）
* 列表与详情数据需一致

---

## 🚫 禁止行为

AI 不允许：

* 做成“组件展示页”
* 每个页面只放1-2个组件
* 无交互逻辑
* 无状态变化
* 无业务语义（只是 UI 拼接）

---

## ✅ 验收标准（核心）

最终结果必须让人感觉：

👉 **“这个项目我可以直接改成自己的业务系统”**

而不是：

👉 “这是个组件 demo”
